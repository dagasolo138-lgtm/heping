# 《生灵》机制优化与开发规划

## 总目标

《生灵》要成为一个可长期运行、可解释、会自然产生历史的规则驱动世界。人物传记、关系变化、史书和未来 AI 对话必须来自已经发生的世界事实。

## 机制原则

1. **先约束，再评分。** 路线、库存、季节、资源和任务名额先通过硬校验，再进入效用评分。
2. **所有占用都要预留。** 任务名额、资源物件、营地容量、工地建材、维护材料和工具均使用可释放预留。
3. **所有变化都要有事实链。** 库存、耐久、劳动、失败和替换进入流水、生命周期与日报。
4. **个人与聚落分账。** 人物背包、家庭储备、公共库存和工地库存逐步分离。
5. **世界时间独立于画面。** 固定 tick 推进世界；UI 只观察，不参与结算。
6. **危机必须有物质原因。** 饥荒、生产衰退和冲突由资源、工具、劳动、运输与分配约束产生。
7. **AI 只负责表达。** AI 不得决定行动或修改世界状态。
8. **长期运行先于扩展。** 新机制必须通过守恒、存档、确定性回放与长跑。

### v0.27 生存经济 ✅

#### 动态目标库存

```text
目标周期 = 未来 3 日
有效库存 = 营地现货 + 人物背包 + 在途资源 - 已承诺资源
```

目标读取人口、季节、气温、天气、腐败、储存保护、篝火、建造和工具维护需求。

#### 劳动成本

路线、负重、地形、道路、天气、精力、技能、行动强度和工具共同决定预计耗时与能耗。

#### 工具与耐久

- 石斧、搬运篮、简易农具和石镐属于公共资产。
- 工具效果进入劳动成本。
- 同一工具不能被两个任务同时占用。
- 成功任务产生磨损；失败和取消只释放占用。

#### 统一资源流水

```text
production / transfer / consumption / fuel
construction / spoilage / wear / repair / replacement
```

每笔记录保存时间、账户、物品、数量、类别、原因、人物、任务、预留和元数据。

#### 每日经济与生命周期

- 日报保存期初、期末、生产、流出、劳动、拒绝、目标库存、瓶颈和账实差异。
- 任务记录 `active / completed / cancelled / failed`。
- 跨午夜使用 `carryIn / carryOut`；真实超时才进入 `overdue`。
- 两阶段建材运输保持单一任务 ID。

#### 长期稳定性与存档连续性

- 第 60 日 batch 1、5、10 完整摘要一致。
- 第 120 日长跑每 15 日检查任务、预留、工具、流水、日报、内存和吞吐。
- 成功读档采用 `cancel-and-replan`。
- 失败读档恢复长期状态、行动代理、路径、进度、统一预留、工具占用、日志和诊断。

## v0.28 工具维护经济

### v0.28.0 维修需求事实层 ✅

- `healthy / worn / critical / broken` 工具状态。
- 稳定且可持久化的维修需求。
- 需求记录目标耐久、材料、工时、技能、请求时间、原因和优先级。
- 工具 schema v2，兼容 v1 存档。

### v0.28.1 真实维修任务 ✅

```text
工具磨损
→ repair 需求
→ 预留目标工具和材料
→ 村民执行劳动
→ 扣材料、恢复耐久
→ 生命周期、人物事实、repair 流水、日报
```

- 紧急生存优先于维修。
- 紧急维修优先于普通生产。
- 维护并发上限为一。
- 执行前再次校验需求、工具、地点、预留和材料。
- 材料中途消失时整单失败，不吞料、不改耐久。
- 失败读档精确恢复活动维修任务及预留。
- 普通 CI、真实移动 Chromium、第 60 日多批次和第 120 日审计已通过。

### v0.28.2 工具替换与最低公共保障 🚧

当前实现：

- 工具 schema v3：`generation / repairsSinceReplacement / wearSinceReplacement`。
- 维护需求 schema v2：`repair / replace`。
- 同一代完成两次维修后，再次磨损会进入替换。
- 本代累计磨损达到最大耐久 2.5 倍时也会进入替换。
- 新增 `replaceTool` 真实劳动任务。
- 替换使用更多木料与工时，成功后进入下一代并清零本代计数。
- 替换材料和耐久恢复进入独立 `replacement` 流水及日报对账。
- 石斧、搬运篮和简易农具最低保障均为一件可用工具。
- 保障缺口把恢复任务提升到最高优先级。
- 未启动需求进入目标库存缺口；已预留材料进入承诺库存。
- v1/v2 工具存档迁移到 v3 时不会凭空触发替换。
- 活动替换任务纳入失败读档检查点。

封版门槛：

- 全量单元、集成、替换闭环、守恒与存档回滚测试通过。
- 真实移动 Chromium 与生产构建通过。
- 第 30/60 日旧基线保持不变。
- 第 60 日 batch 1/5/10 在代际工具世界中保持一致。
- 第 120 日无未保护保障缺口、幽灵维护预留、账实差异或生产锁死。
